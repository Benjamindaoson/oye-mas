# 主编排原理与流程

**版本**:v3.0(对应 ADR-001-rev / ADR-004 / ADR-009~016)
**日期**:2026-05-05
**面向**:产品 / 设计 / 新工程师入职阅读
**地位**:讲流程图、概念、用户感知。**不写代码**;实现细节请看 [主编排 Agent 实现指南](../2_工程实现/主编排 Agent 实现指南.md)。

**v3.0 关键变更**:
- ✨ Agent 编号反向(ADR-001-rev):Agent 2=文档 / Agent 3=图 / Agent 4=影音
- ✨ 子模块从 7 升级到 **9 个**(新增 HITL 网关管理器 + Agent 互动编排器)
- ✨ 三种模式同群内切换(ADR-014)取代两种独立群
- ✨ 2 个支持 Agent(HR + 财务经理,ADR-013)接入主会话
- ✨ Agent 拟人化编排(表情 + 状态 + 互动对话,ADR-015)

---

## 一、主编排是什么

主编排 Agent(在用户面前叫"**总裁助理**")是「有了」的**大脑**和**唯一调度者**。

特征:

- **不直接产出内容**(不写文案、不画图、不剪视频)
- 每个用户只有一个,跟着用户走(主会话 + 所有群都是同一个)
- 是**唯一派活方**——Agent 1-4 之间不互相派活
- 调用最频繁,成本最低:每个任务调用 5-15 次,主力 deepseek-v4-flash,备 Claude Haiku 4.5

---

## 二、整体流程

```
用户消息进入群
    ↓
[1] 意图理解
    ↓
[2] Skill 匹配
    ↓
[3] 输入校验
    ↓
缺信息?
    ├─ 是 → [4] 意图澄清(选择题) → 用户回答 → 回到 [3]
    └─ 否 → [5] 任务编排
              ↓
            [6] 派活给 Agent 1-4
              ↓
            执行中遇到用户消息?
              ├─ 是 → [7] 中断处理 → 派发到对应处理器
              └─ 否 → 继续执行
```

主编排内部分 **9 个子模块**,每个模块独立上下文,各管各的。

---

## 三、子模块概览

| # | 子模块 | 是否调 LLM | 主要职责 |
|---|--------|-----------|----------|
| 1 | 意图理解器 | ✅ 调 | 自然语言 → 结构化意图 JSON |
| 2 | Skill 匹配器 | 99% 不调 | 三层检索(SQL → 向量 → LLM) |
| 3 | 输入校验器 | ❌ 纯程序 | 字段是否齐全 |
| 4 | 澄清生成器 | 大部分不调 | 程序生成选择题 + LLM 生成内容版本 |
| 5 | 任务编排器 | ❌ 纯程序 | Jinja2 渲染 prompt + 打包 AgentTask |
| 6 | 中断处理器 | ✅ 调 | 8 类中断分类与派发(含 V1 中断 C 回滚)|
| 7 | **三模式管理器**(v3.0)| ✅ 调 | Plan / Ask / Auto 同群内切换 + brief 维护 |
| 8 | HITL 网关管理器 | 半 | 在 Skill 工作流的关键节点暂停,等待用户决定 |
| **9** | **Agent 互动编排器**(v3.0)| ✅ 调 | 派活时统一编排 Agent 间互动消息(ADR-015)|

**关键设计**:

- 主编排不是"一个大 LLM 调用",是 5+ 个独立小 LLM 调用 + 大量纯程序处理
- 每次 LLM 调用上下文 < 1500 tokens
- Skill 不会全文进入 LLM 上下文(只有元数据)
- Prompt 渲染是纯程序操作,不需要 LLM

## 四、子模块 1:意图理解

### 4.1 职责

把用户的自然语言变成结构化的"意图 JSON"——后续所有模块都基于这个 JSON 工作。

### 4.2 输出 schema(简化版)

```json
{
  "intent_type": "create_task | modify_task | query | feedback | chitchat | meta | back_to_discuss | ready_to_work",
  "domain": "text | image | video | document | mixed | null",
  "scenario": "anti_fraud | ecommerce_detail | ... | null",
  "entities": {"字段名": "字段值或 null"},
  "confidence": 0.0
}
```

**字段含义**:

- `intent_type`:用户想做什么类型的事
- `domain`:任务所属领域
- `scenario`:具体场景(对应 Skill 的标签)
- `entities`:从消息中提取的实体值;没提到的字段是 `null`
- `confidence`:意图理解的置信度;< 0.7 → 触发澄清,让用户确认意图

### 4.3 例子

**例 1**(明确意图):

输入:"做反诈视频,2026 投资理财案例"

输出:`scenario=anti_fraud, entities={year:2026, fraud_type:investment, audience:null}, confidence=0.95`

**例 2**(模糊意图):

输入:"我想做点东西"

输出:`domain=null, confidence=0.4` → 触发澄清:"你想做什么? [视频] [图片] [文章] [文档]"

**例 3**(中断意图,任务进行中):

输入:"等等,案例换成电信诈骗"

输出:`intent_type=modify_task, entities={fraud_type:telecom}, confidence=0.93` → 触发中断处理(C 类:修改参数)

## 五、子模块 2:Skill 匹配

### 5.1 职责

根据意图 JSON,从 Skill 库找到最合适的 Skill。

### 5.2 三层检索(99% 不调 LLM)

| 层 | 方式 | 性能 | 成本 | 命中条件 |
|----|------|------|------|----------|
| L1 | 关键词 SQL(jieba 分词 + `keywords @>`)| 5-30ms | 0 | match_count ≥ 2 |
| L2 | pgvector HNSW(BGE-M3 1024 维)| 30-50ms | 0 | similarity ≥ 0.85 |
| L3 | LLM 兜底(Top 10 候选简介)| 600-800ms | $0.0002 | confidence ≥ 0.7 |

### 5.3 找不到匹配怎么办

三层都失败 → 返回 Top 3 候选给用户确认:

```
我不太确定你想做什么,是这几个吗?
[反诈视频制作] [小红书爆款笔记] [都不是]
```

- 用户选其中一个 → 进入下一步
- 用户选"都不是" → 让用户用自然语言描述,转人工或建议建群

### 5.4 当前群的限制

如果用户在"反诈视频群"里说"做小红书笔记"——这个群没有装备小红书 Skill。

**处理方式**:

> "这个群是反诈视频专用,你想做小红书笔记的话,我帮你新建一个群?[新建小红书群] [继续在这个群做反诈视频] [取消]"

## 六、子模块 3:输入校验

### 6.1 职责

检查用户提供的字段是否够启动 Skill。**纯程序逻辑,不调 LLM。**

### 6.2 字段优先级

```
优先级 1:用户消息里提到了                  (intent.entities)
优先级 2:偏好库可信度足够(≥ 1.0)         (user_preferences)
优先级 3:Brief 字段(讨论群衍生工作群时)   (context_pool.brief.字段)
优先级 4:选填字段用默认值                  (field.default)
缺失:必填且找不到 → 加入 missing_fields
```

### 6.3 偏好可信度阈值

| 用户行为 | 偏好可信度 | 处理 |
|----------|-----------|------|
| 第 1 次选某值 | 0.3 | 不影响下次询问 |
| 连续 2 次同选 | 0.7 | 仍要问,但选项默认值用偏好值 |
| 连续 3 次同选 | **1.0** | **自动套用,不再问** |

只有可信度 ≥ 1.0 才自动套用。

## 七、子模块 4:意图澄清

### 7.1 核心原则

**永远选择题,不超过 5 轮,不让用户填空。**

### 7.2 4 种澄清形式

#### 形式 1:单字段单选(纯程序)

最常见。直接 3-4 个按钮:

```
受众? [城市老人] [农村老人] [都覆盖]
```

#### 形式 2:多字段合并(纯程序)

缺多个字段时,打包成一个气泡一次问完:

```
确认 2 件事:
受众? [城市老人] [农村老人] [都覆盖]
时长? [30 秒] [60 秒] [90 秒]
```

适用于缺 2-3 个字段的情况。超过 3 个,拆 2 轮。

#### 形式 3:图片对比(纯程序,加载预置图)

适用于视觉风格类字段。3 张参考图缩略图供选:

```
你想要哪种风格?
[图 1: 简约风] [图 2: 写实风] [图 3: 卡通风]
```

参考图来自 OSS 中预置的样本图。

#### 形式 4:内容版本对比(调 LLM)

适用于"开头风格"、"标题选择"等需要看到具体内容才能选的情况。**实时调 Agent 1 生成 3 个版本**(走标准的 Agent 派活流程,不是主编排自己调)。

```
这 3 个开头你喜欢哪个?
[版本 A: "你信吗?这位老人..."]
[版本 B: "60 万,就这样没了..."]
[版本 C: "退休工资全部被骗..."]
```

注意:这种澄清需要等 Agent 1 生成,有 3-5 秒延迟。给用户"AI 正在生成选项..."的提示。

### 7.3 兜底机制

- **每个字段都有默认值**:用户超时未答(60 秒),自动用默认值继续
- **底部一个不显眼的"自定义"按钮**:点了进入自由输入模式(适合预设选项不够用的情况)
- **不超过 5 轮**:超过 5 轮还有缺失,用默认值或偏好兜底,不继续问

## 八、子模块 5:任务编排

### 8.1 职责

把"已选定的 Skill + 已填满的字段"变成"派给具体 Agent 的可执行任务"。**纯程序处理,不调 LLM。**

### 8.2 编译过程(只做一次,缓存到 Redis)

```
Skill YAML
    ↓
Parser 解析为内存对象
    ↓
为每个 step 创建 LangGraph Node
    ↓
根据 depends_on 创建 Edges
    ↓
编译为可执行的 StateGraph
    ↓
绑定 PostgreSQL checkpointer
    ↓
缓存到 Redis(key = skill_id + version)
```

### 8.3 Prompt 渲染(纯程序)

```
context = {
    用户字段(collected_fields),
    上游产物(按 step_id 索引,从 OSS 按需加载)
}
rendered_prompt = Jinja2.render(template, **context)
```

**例**:Skill 模板 `基于 {{research.output}} 撰写脚本,针对 {{受众}}` → 渲染后变成完整文本,直接发给 Agent 1。

### 8.4 派给哪个 Agent

由 Skill 的 `agent` 字段直接指定(`agent_1` / `agent_2` / `agent_3` / `agent_4`),**主编排不需要"判断"**。

> ⚠️ **ADR-002 锁定**:Agent 之间不主动跨调。任何"我需要别的 Agent 帮忙"的场景,必须由 Skill YAML 显式拆为独立 step,LangGraph 自动调度。

### 8.5 并行执行

LangGraph 自动识别可并行节点(没有相互依赖的)。例:research 完成后,script 和 image_process 同时启动(都依赖 research,但相互无依赖)。

**不需要 Skill 定义里写"哪些可以并行"**,LangGraph 拓扑调度自动处理。

## 九、子模块 6:中断处理

### 9.1 触发时机

任务执行中(`state.status == "executing"`),用户在群里发新消息。

### 9.2 8 类中断

| 类型 | 含义 | 处理方式 | V1 |
|------|------|----------|----|
| A | 补充信息 | graph.update_state + invoke 继续 | ✓ |
| B | 微调当前 | 取消当前节点 + 改 prompt + 重启 | ✓ |
| C | 修改参数 | 依赖图回滚 + 重做受影响步骤 | V2 |
| D | 改方向 | 当前 graph 标 abandoned + 创建新 graph | V2 |
| E | 暂停 | task.status = paused | ✓ |
| F | 取消 | task.status = cancelled,24h 后清理 | ✓ |
| G | 闲聊 | 不动 graph,直接回复 | ✓ |
| H | 反馈 | 写事件流,可能转 B | ✓ |

### 9.3 关键约束

- **置信度 < 0.7 时不直接处理**,先让用户确认意图("你是想暂停这个任务吗?")
- **永远先回复用户**(让用户知道收到了),再操作 graph
- **中断处理本身不能被中断**(不然永远处理不完)

---

## 九.5 子模块 8:HITL 网关管理器(v2.0,ADR-010)

### 9.5.1 职责

在 Skill 工作流的关键节点(脚本完成 / 配图完成 / 视频初剪)**主动暂停**,等待用户审核或调整,确保 hero 交付物质量。

### 9.5.2 触发条件

Skill YAML 在 step 上声明 `hitl_gate`:

```yaml
- step_id: script_versions
  hitl_gate:
    type: version_select         # version_select / quality_review / final_approval
    timeout_seconds: 600         # 10 分钟未响应自动选 default
    auto_approve_if_user_offline: false
```

主编排执行到该 step 后:

1. 执行完 step → 拿到产物
2. **暂停 LangGraph state**(`status = waiting_hitl`)
3. WS 推 `hitl_gate_opened` → 前端显示审核组件
4. 等待用户响应

### 9.5.3 用户的 4 种响应

| 选项 | 行为 | 转后续处理 |
|------|------|----------|
| **接受** | 推进到下一 step | 任务正常继续 |
| **微调当前** | 重做当前 step,带用户修改提示 | 转中断 B |
| **回到第 N 步重做** | 回滚到指定 step | **转中断 C(V1 必须实现)** |
| **取消** | 终止任务 | 转中断 F |

### 9.5.4 短视频的 3 个 HITL gate

```
脚本生成 → 【脚本审核】(version_select)
画面素材 → 【画面审核】(quality_review)
视频合成 → 【终审】(final_approval,含回滚选项)
```

### 9.5.5 兜底

- 用户超时(默认 10 分钟)→ 选 default(`auto_approve_if_user_offline: false` 时不自动通过,而是发通知后无限等待)
- 用户离线 → 任务标 `paused`,WS 重连时弹"继续审核"提示

## 十、子模块 7:模式管理(讨论 / 干活)

### 10.1 职责

根据当前群的 `mode`,派发到不同的处理流程,并维护讨论群的 brief。

### 10.2 派发逻辑

```
conversation.mode 是?
    ├─ main_session  → 接受元操作,弹出"模式选择"
    ├─ discuss       → 不走 Skill,自由对话 + brief 沉淀
    ├─ work          → 走 Skill 工作流(用 brief 预填字段)
    └─ private_chat  → 跳过编排,直接转给单 Agent
```

### 10.3 Brief 维护

讨论群里每条用户消息都(防抖批量地)更新 ContextPool 的 brief。

**触发条件**(满足任一即更新):

1. 用户停顿 5 秒(打字结束防抖)
2. 累计 3 条新消息未更新
3. 用户主动点 [总结一下]
4. 总裁助理被 @ 时强制更新

**完成度评估**:`完成度 ∈ [0.0, 1.0]`,达到 0.8 → 总裁助理主动建议"拉工作群开始制作"。

详见 [两种模式产品定义](两种模式产品定义.md) + [两种模式技术实现](../2_工程实现/两种模式技术实现.md)。

## 十一、整体流程示例

用户在反诈视频群说:"做反诈视频,2026 投资理财"

```
T+0       用户发消息
T+0.1     [意图理解] deepseek-v4-flash,300ms,1500 tokens
          输出:scenario=anti_fraud, entities={year:2026, fraud_type:investment}, confidence=0.95
T+0.4     [Skill 匹配] L1 关键词命中,5ms,0 tokens
          输出:skill_id=anti_fraud_video, confidence=0.95
T+0.41    [输入校验] 纯程序,1ms
          输出:filled={年份, 骗局类型, 时长}, missing=[受众]
T+0.42    [澄清生成] 纯程序(单字段单选)
          输出:选择题"受众? [城市老人] [农村老人] [都覆盖]"
T+0.5     推送给前端

[等用户选择...]

T+10      用户点 [城市老人]
T+10.1    [输入校验] 重新校验,字段全齐
T+10.15   [任务编排] 加载 Skill workflow,编译为 LangGraph
T+10.2    主编排回复:"任务开始,研究员先调研..."
T+10.3    启动 research 节点,渲染 prompt,提交 agent_tasks:text 队列
[Agent 1 开始执行,用户看到右栏更新...]
```

## 十二、性能与成本汇总

| 子模块 | LLM 调用 | 上下文 | 延迟 | 成本/任务 |
|--------|---------|--------|------|----------|
| 意图理解 | deepseek-v4-flash | 1500 tokens | 300ms | $0.0002 |
| Skill 匹配 | 99% 不调,1% 调 | 0 / 1500 | 5-30ms / 600ms | $0 / $0.0002 |
| 输入校验 | 不调 | 0 | 1ms | $0 |
| 澄清生成 | 大部分不调 | 0 / 1000 | 1ms / 3-5s | $0 / $0.001 |
| 任务编排 | 不调 | 0 | 50-200ms | $0 |
| 中断处理 | deepseek-v4-flash | 1500 tokens | 300ms | $0.0002 |
| 模式管理(brief) | deepseek-v4-flash | 1000 tokens | 300ms | $0.0002 |

**主编排单任务总成本**:$0.001 - $0.003(15 次调用)
**主编排单次调用响应**:< 1 秒(满足实时性)

## 十三、关键判断回顾

1. 主编排内部分 **7 个子模块**,每个独立上下文
2. 大部分模块**不调 LLM**(纯程序),只有意图理解、Skill LLM 兜底、中断分类、Brief 维护调 LLM
3. Skill 渲染是 **Jinja2 模板渲染**,纯程序
4. State 用**引用**,不存全文(产物 = artifact_id + OSS URL)
5. Agent 派活通过**消息队列**,Agent 之间不互相调用(ADR-002)
