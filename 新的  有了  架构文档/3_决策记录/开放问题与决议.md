## ADR-001:Agent 编号分工

**日期**:2026-05-05

| 产品文档 + Excel + 主编排实现指南示例 | 设计师(图)| 影音师(视频/音频)| 文档专员(Office)|

| 编号 | 角色名(v2.0 ADR-001 已废)| 能力域 | 队列名 |
|------|--------|--------|--------|
| Agent 1 | 研究员 / 文案师 | 文字与语言 | `agent_tasks:text` |
| ~~Agent 2~~ | ~~设计师~~ | 图像 | `agent_tasks:image` |
| ~~Agent 3~~ | ~~影音师~~ | 视频与音频 | `agent_tasks:av` |

|------|--------|--------|

4. **新分法没给出迁移理由**——4 个分任务 Agent 指南只是默默换了

## ADR-002:Agent 不主动跨调其他 Agent

**状态**:已采纳
**日期**:2026-05-05

### 背景

历史文档存在矛盾:

- **多处文档反复强调**:"主编排是唯一调度者,Agent 之间不互相 @、不直接对话"
- **4 个分任务 Agent 指南**(前版)的 `pptx_create_handler` 却写了 `call_other_agent("agent_1", ...)` 自己跨调

如果允许 Agent 自己跨调:

- 主编排失去全局视图,无法正确分配中断 / 计费 / 监控
- Agent 之间可能形成环形依赖、死锁
- 故障会通过跨调链路传染

### 决定

**Agent 不主动跨调其他 Agent。**

- Agent 内部只能调自己的工具/模型(同进程)
- 任何"我需要别的 Agent 帮忙"的场景,必须**在 Skill YAML 里声明为独立 step**,由主编排 LangGraph 编排
- `call_other_agent` 函数从代码库删除,取而代之是抛 `NotImplementedError`(防御性编程)

### 范式对比

**❌ 错误(已删除)**:

```python
# Agent 4 内部
async def pptx_create_handler(task, model):
    outline = await call_other_agent("agent_1", "structured_writing", ...)  # 跨调
    images = await call_other_agent("agent_2", "image_generate", ...)        # 跨调
    return assemble_ppt(outline, images)
```

**✅ 正确(Skill YAML 显式拆步)**:

```yaml
workflow:
  - step_id: ppt_outline
    agent: agent_1
    task_type: structured_writing
  - step_id: ppt_images
    agent: agent_2
    depends_on: [ppt_outline]
    task_type: image_generate
  - step_id: ppt_assemble
    agent: agent_4
    depends_on: [ppt_outline, ppt_images]
    task_type: pptx_assemble    # 仅组装,不创作内容
```

### 落地状态

- [x] 4 个分任务 Agent 实现指南 §1.8 加入 `call_other_agent` 防御性 stub
- [x] `pptx_create` task_type 改名 `pptx_assemble`(语义变化:仅组装)
- [x] PPT Skill YAML 显式拆为三步
- [x] 反诈视频 Skill 已是显式拆步形态(无遗留问题)

## ADR-006:Brief 防抖批量更新策略

**状态**:已采纳
**日期**:2026-05-05

### 背景

讨论群每条用户消息都更新 brief,会产生大量 LLM 调用。
- 5 分钟讨论 ≈ 10-15 条消息 ≈ 10-15 次 LLM 调用 ≈ $0.01

但如果只在用户主动点 [总结一下] 时更新,brief 完成度无法实时驱动"主动建议工作群"的逻辑。

### 决定

**三层防抖批量触发机制**:

满足任一条件即更新:

1. **5 秒防抖**:用户停顿 5 秒(打字结束)
2. **3 条阈值**:累计 3 条新消息未更新时强制触发
3. **主动触发**:用户点 [总结一下] / 总裁助理被 @
4. **强制触发**:讨论群关闭前

### 理由

- 5 秒防抖捕获"自然对话节奏"——用户表达完一段话再更新
- 3 条阈值兜底——防止用户连续打字时永远不更新
- 实测成本:5 分钟讨论触发 3-5 次 brief 更新 ≈ $0.003,可接受

### 落地状态

- [x] 主编排实现指南 §9.5 提供调度器实现
- [x] 两种模式技术实现 §7.1 提供完整实现

## ADR-007:模型路由分两层(L1/L2)

**状态**:已采纳
**日期**:2026-05-05

### 背景

主编排实现指南说"主力 deepseek-v4-flash";4 个分任务 Agent 指南给了详细 task_type → 模型映射表。两者容易让人误解"是不是全系统都用 Flash"。

### 决定

**两层路由表**(LiteLLM 配置内分别维护):

**L1 编排层路由**(主编排子模块用):

- 意图理解 → deepseek-v4-flash / Haiku
- Skill LLM 兜底匹配 → 同上
- 中断分类 → 同上
- Brief 更新 → 同上
- 共同点:轻量决策,要求快、便宜、JSON 输出

**L2 执行层路由**(4 个 Agent 用):

- long_writing → Kimi K2 / DeepSeek-v4-pro
- image_generate(真实)→ GPT-Image-2
- text_to_video → Veo-3
- TTS → Volcengine-TTS
- 共同点:按 task_type 选最强模型,接受较高延迟

详细配置见 [模型路由表](../4_附录/模型路由表.md)。

### 理由

- 两层路由的健康度 / 降级策略独立,互不干扰
- 主编排的低延迟要求(<1s)与 Agent 的高质量要求(分钟级)矛盾,分开维护
- 成本审计可分别统计

### 落地状态

- [x] 模型路由表文档已建立 L1/L2 分章节
- [x] 主编排实现指南所有 LLM 调用走 L1
- [x] 4 个 Agent 实现指南所有 LLM 调用走 L2

## ADR-008:取消"通用群"独立概念

**状态**:已采纳
**日期**:2026-05-05

### 背景

产品文档 V1 范围曾出现:

- 群形态 1:**通用群**(装备全部 4 个 Agent)
- 群形态 2:**专门群**(装备特定 Skill)

而两种模式文档同时引入:

- **讨论群**(讨论模式)
- **工作群**(干活模式)

"通用群"和"讨论群"装备的 Agent 重叠,用户感知层模糊不清。

### 决定

**取消"通用群"作为独立概念,合并到"工作群"分类**:

```
群形态:
├─ 讨论群(mode=discuss)
│  └─ 装备总裁助理 + 研究员 + 设计师
│
├─ 工作群(mode=work)
│  ├─ 通用工作群(skill_id=null,全 4 个 Agent)
│  └─ 专门工作群(skill_id=xxx,装 Skill 包)
│
└─ 主会话(mode=main_session)
   └─ 只有总裁助理
```

用户感知层简化:**讨论 / 工作 二选一**。

### 理由

- "通用 vs 专门"是工程概念,用户不需要理解
- 新用户只需要回答一个问题:"先聊聊还是直接做"
- 简化后偏好沉淀也简单(只追踪 default_mode)

### 落地状态

- [x] 产品功能与交互文档 §6 已重写为两种模式
- [x] Excel 功能清单 v3 已对齐(条目 71 改为"通用工作群"、72 改为"专门工作群")
- [x] 两种模式产品定义文档已落地

## ADR-009:MCP 协议作为工具集成唯一标准

**状态**:已采纳
**日期**:2026-05-05

### 背景

v1.0 架构里,Agent 1 的 Tavily/Playwright、Agent 2 的 PIL/rembg、Agent 3 的 FFmpeg/Whisper、Agent 4 的 python-pptx 等工具**都直接写在 handler 里**(直连 SDK)。

到 2026 年,**Anthropic 的 Model Context Protocol(MCP)已经成为 agent 工具集成的事实标准**,Claude Code、Cursor、Anthropic Claude Desktop、ChatGPT Operator 等都已采用,社区 MCP server 生态(Linear、Notion、Slack、GitHub、Figma 等)蓬勃发展。

**继续用 v1.0 的"handler 直连工具"会带来 4 个问题**:

1. 接入新工具成本高(每个 Agent 都得改 handler 代码)
2. 不能复用社区 MCP server 生态
3. Skill 创作者飞轮(V1.5)难做——创作者无法用标准协议引用工具
4. 调试 / 审计 / 限流不统一

### 决定

**所有外部工具一律以 MCP server 形式暴露,Agent handler 作为 MCP client 调用。**

#### 实施细则

1. **`mcp_servers/` 目录**统一管理 MCP server 实现
2. V1 必上线 7 个 MCP server:
   - `mcp-search`(web_search / web_fetch)
   - `mcp-image-tools`(bg_remove / enhance / concat_long / quality_check)
   - `mcp-video-tools`(compose / extract_frames / subtitle_align)
   - `mcp-audio-tools`(tts / asr / bgm_match)
   - `mcp-document-tools`(pptx / xlsx / docx / pdf)
   - `mcp-oss`(upload / download / sign_url)
3. V1.5 增加 `mcp-platform-publish`(抖音 / 视频号 / 小红书发布 API)
4. **图像 / 视频 / 文本生成模型仍走 LiteLLM**(模型生成不属于"工具"范畴)
5. Skill YAML 引用:`mcp://search/web_search`
6. Agent handler 调用:`await mcp_client.call_tool(server="...", tool="...", arguments={...})`

#### 例外

- **生成式模型走 LiteLLM**(GPT-Image-2 / Veo-3 / Volcengine TTS)。这些是模型而非工具。
- LiteLLM 本身不通过 MCP(它是模型路由,不是工具)。

### 落地状态

- [x] 主提示词 §3.1.5 加入 MCP 章节
- [x] 仓库结构加入 `mcp_servers/`
- [x] 4 个分任务 Agent 实现指南(待更新到 v2.0)
- [x] Skill YAML 模板支持 `mcp://` URI 引用
- [ ] 实际 MCP server 实现(Sprint 1)

## ADR-010:Hero 交付物必有 HITL gate(中断 C 提到 V1)

**状态**:已采纳
**日期**:2026-05-05

### 背景

v1.0 设计里,视频任务是 5-10 分钟的长流程,但**没有人在回路(human-in-the-loop)审核**。中断 C(修改参数,触发依赖图回滚)被推到 V2。

**问题**:

1. 视频生成模型(Veo-3 / Seedance-2 / Kling-2)在某些场景下成功率仍 70-85%
2. 用户花 8 分钟等一个不满意的视频,只能 [追问] 或 [改一下](中断 B,只重做当前 step),无法回到第 2 步重做脚本
3. 这意味着 V1 视频成功率必须 ≥ 90% 才能 ship,**而当前任何模型都做不到**
4. SV 同行(Synthesia / HeyGen)都在 hero 节点做 HITL,这是经过验证的成功公式

### 决定

**Hero 交付物(视频 / 长图 / 长文档)必须在关键节点暂停,等待用户审核,且 V1 必须支持中断 C(回滚)。**

#### 实施细则

1. **Skill YAML 支持 `hitl_gate` 字段**:

```yaml
- step_id: script
  agent: agent_1
  task_type: short_video_script
```

## ADR-011:数据飞轮 4 类信号沉淀必做

**状态**:已采纳
**日期**:2026-05-05

### 背景

v1.0 的"用户偏好"机制是简单计数器(连续 3 次同选 → confidence=1.0)。这是 2018 年的水平,放在 2026 年:

- 没有用户使用数据反哺 LLM
- Skill 是平台/创作者写死的,不会随用户群体进化
- 失败任务的经验没有沉淀
- "数据飞轮"是 SV 投资人评判 AI 产品防御力的核心维度

### 决定

**每个任务完成必须产生 4 类训练信号**:

#### 信号 1:工作流完整轨迹 → Qdrant RAG

每次任务的完整 trace(用户输入 / 各步骤产物 / 用户在 HITL gate 的选择 / 最终满意度)进 Qdrant 向量库,供后续 RAG 引用。

```python
await qdrant.upsert(
    collection="workflow_traces",
    points=[Point(
        id=task.id,
        vector=embedding,
        payload={
            "user_id": ..., "skill_id": ...,
            "duration_ms": ..., "user_satisfaction": 1-5,
            "trace_oss_ref": "..."
        }
    )]
)
```

用途:新任务开始时,主编排可以 RAG 检索"类似用户类似任务以前怎么做的",作为 prompt 增强。

#### 信号 2:用户偏好向量 → bge-base-zh-v1.5(256 维)

每次用户做选择(澄清回答、HITL 决定),更新 user_preferences.preference_vec。

升级前:`{value: 城市老人, confidence: 0.7}`(标量)
升级后:256 维向量,捕获"风格偏好 / 节奏偏好 / 受众偏好 / 平台偏好"4 维度

用途:新任务的字段默认值动态排序,而不是硬编码 default。

#### 信号 3:失败 → Reflexion → prompt 改进候选

任务失败 / 用户负面反馈 / HITL 拒绝 → 自动跑 Reflexion pipeline:

```python
async def reflexion_pipeline(task, failure):
    """
    用 Claude Sonnet 4.6 分析失败原因,提出 prompt 改进建议。
    写入 prompt_improvement_candidates 表,人工审核后合入 prompts.py。
    """
    analysis = await claude_sonnet.complete(
        prompt=f"任务失败,prompt: {original_prompt}, 失败模式: {failure}, 建议如何改进 prompt"
    )
    await db.insert("prompt_improvement_candidates", ...)
```

#### 信号 4:高质量任务 → 自动起 Skill 草稿(创作者飞轮 V1.5)

满意度 ≥ 4 + 流程偏离平台预置 Skill → 自动生成 Skill YAML 草稿,推送给用户:**"这个工作流不错,要发布为 Skill 吗?"**

这是创作者飞轮的种子机制,**让 V1.5 的 Skill 市场不需要冷启动**。

### 实施细则

- 新建 `app/services/flywheel.py`(信号发射器)
- 新增表:`workflow_traces`、`prompt_improvement_candidates`、`skill_drafts`
- `user_preferences` 加列:`preference_vec VECTOR(256)`
- 每次任务完成的最后一步必经 `flywheel.emit_*`(在 `task_compiler.py` 末尾 hook)
- CI 检查:任何修改 task 完成路径的 PR 必须保留信号沉淀,否则失败

### 隐私合规

- 用户协议明确:"工作流数据用于改进系统,可通过设置关闭"
- 关闭时,信号 1/3 不沉淀,信号 2/4 仍沉淀(因为属于个人服务)
- 工作流轨迹脱敏后可用于训练(去除用户 ID、上传素材)

### 落地状态

- [x] 主提示词 §3.4.4 数据飞轮信号沉淀
- [x] 5 张新表 schema 设计
- [x] 仓库结构 `flywheel/` pipeline
- [ ] flywheel.py 实现(Sprint 4)
- [ ] Reflexion pipeline 实现(Sprint 4)

## ADR-014:三种工作模式同群内切换(Plan / Ask / Auto)

**状态**:已采纳(取代 ADR-008)
**日期**:2026-05-05

### 背景

ADR-008(v2.0)曾设计**讨论群 / 工作群两种独立群**,共享 ContextPool。

产品方在最终版改为:**三种工作模式是同一个群内的工作方式,可以中途切换,不是建不同的群**:

| 模式 | 英文 | 行为 |
|------|------|------|
| **讨论模式** | Plan | 聊需求 / 想方向,不动手做 |
| **询问模式** | Ask | 用户问问题,AI 只回答,不创建任务 |
| **自动模式** | Auto | AI 团队自主执行 Skill 工作流 |

### 决定

#### 数据模型变化

**废弃**:

- `conversation_links` 表(讨论群衍生工作群)
- `conversations.parent_conversation_id`
- `conversations.mode` 枚举里的 `discuss` / `work` 拆分

**新设计**:

```sql
-- conversations 表
mode VARCHAR(20) CHECK (mode IN ('main_session','group','private_chat'))
-- 普通群只有 'group' 一种 mode

-- 新增字段
work_mode VARCHAR(10) CHECK (work_mode IN ('plan','ask','auto'))
-- 工作模式,group 类型 mode 必填,可在群内切换
```

**保留**:

- `context_pools` 仍存在,但用途收窄(brief 维护、参考产物缓存)
- 不再"两群共享 pool";一个群一个 pool

#### 切换 API

```
POST /api/conversations/:id/switch-work-mode
Body: {target: "plan" | "ask" | "auto"}

→ 主编排:
  - 写入消息 "总裁助理:已切到 X 模式"
  - 后续行为按新 mode 走
  - LangGraph state 不重启(模式只改派活策略,不影响进行中的步骤)
```

#### 三模式行为

| 模式 | 主编排行为 |
|------|----------|
| Plan | 不调 Skill,只调研究员/设计师生成参考;brief 后台维护 |
| Ask | 单轮问答,可调任何 Agent 但不持久化 task |
| Auto | 走完整 Skill 工作流,有 HITL gate |

#### Brief 跨模式延续

- 用户在 Plan 模式聊清楚 → 切 Auto 模式时,brief 字段自动填充 Skill inputs
- 同一群内,brief 继承,无需"衍生"机制

### 影响

- **重命名**:`两种模式产品定义.md` → `三种模式产品定义.md`
- **重命名**:`两种模式技术实现.md` → `三种模式技术实现.md`
- **简化**:废弃讨论群衍生工作群的所有逻辑
- 前端:群顶部增加"模式切换 chip",废弃讨论群/工作群双群头像方案

### 落地

- [ ] DB 迁移:加 `work_mode` 列,删 `parent_conversation_id`
- [ ] 主编排:`mode_manager.py` 改三模式分支
- [ ] API:`switch-work-mode` 替代 `derive-work-group` / `back-to-discuss`
