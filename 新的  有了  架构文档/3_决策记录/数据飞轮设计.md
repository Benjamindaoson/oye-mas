## 一、为什么数据飞轮决定 SaaS AI 产品的生死

到 2026 年,foundation model(GPT-5 / Claude 4.6+ / Gemini 2.0 / DeepSeek-V4)能力都很强。**纯应用层的 AI 产品没有 moat**——任何竞品都能复制 prompt + 工作流。

唯一的护城河来自**用户使用数据反哺产品**:

- 用得越多 → 系统越懂你 → 切换成本越高
- 用户群越大 → 工作流多样性越高 → 平台 Skill 越完善
- 失败案例越多 → Reflexion 改进越精准
- 高质量任务越多 → Skill 创作飞轮越快转

**没有数据飞轮的 AI 产品,3 年内被 foundation model 升级吃掉的概率 > 70%**。

## 二、4 类信号

每次任务完成必须产生:

| # | 信号 | 用途 | 存储 |
|---|------|------|------|
| 1 | **工作流完整轨迹** | RAG 检索"类似任务怎么做的" | Qdrant + OSS |
| 2 | **用户偏好向量** | 个性化默认值排序 | Postgres pgvector |
| 3 | **失败 → Reflexion 信号** | 自动改进 prompt | Postgres prompt_improvement_candidates |
| 4 | **高质量任务 → Skill 草稿** | 创作者飞轮(V1.5)| Postgres skill_drafts |

## 三、信号 1:工作流完整轨迹

### 3.1 数据结构

```python
class WorkflowTrace(BaseModel):
    task_id: UUID
    user_id: UUID
    skill_id: UUID
    skill_version: str
    
    # 用户输入
    initial_message: str
    collected_fields: dict
    brief: dict | None              # 来自讨论群
    
    # 中间产物(产物本体存 OSS,这里只存 metadata)
    steps: list[StepTrace]
    
    # 用户在 HITL gate 的决定
    hitl_decisions: list[HITLDecision]
    
    # 中断历史
    interrupts: list[InterruptRecord]
    
    # 最终结果
    duration_ms: int
    cost_usd: float
    final_artifact_ref: str        # OSS URL
    user_satisfaction: int | None  # 1-5,任务结束 24h 内提示用户打分
    
    # metadata
    created_at: datetime
    completed_at: datetime
```

### 3.2 存储

**Qdrant**(向量索引):

```python
collection = "workflow_traces"
vector_size = 1024              # BGE-M3
distance = "Cosine"

await qdrant.upsert(
    collection_name=collection,
    points=[Point(
        id=task.id,
        vector=await bge_m3.embed(trace.summary_for_embedding()),
        payload={
            "user_id": ..., "skill_id": ..., 
            "user_satisfaction": ..., "duration_ms": ...,
            "trace_oss_ref": ...
        }
    )]
)
```

`summary_for_embedding` 是 trace 的精炼摘要(< 300 字),包含:用户初始需求 + 关键澄清选择 + 失败/成功标记。

**OSS**(全文):trace JSON 完整存 `oss://traces/{user_id}/{task_id}.json`

**Postgres `workflow_traces` 表**(索引):

```sql
CREATE TABLE workflow_traces (
    id UUID PRIMARY KEY,
    task_id UUID UNIQUE REFERENCES tasks(id),
    user_id UUID REFERENCES users(id),
    skill_id UUID REFERENCES skills(id),
    skill_version VARCHAR(20),
    duration_ms INTEGER,
    cost_usd DECIMAL(10,6),
    user_satisfaction SMALLINT,
    failure_reason TEXT,
    trace_oss_ref TEXT,
    qdrant_point_id TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_trace_user_skill ON workflow_traces(user_id, skill_id);
CREATE INDEX idx_trace_satisfaction ON workflow_traces(user_satisfaction) WHERE user_satisfaction IS NOT NULL;
```

### 3.3 用途:RAG 增强主编排

新任务开始时,主编排可以检索相似历史:

```python
async def enhance_intent_with_history(intent: Intent, user_id: UUID):
    """新任务的意图理解可以参考历史相似任务"""
    
    # 检索同用户历史
    user_history = await qdrant.search(
        collection_name="workflow_traces",
        query_vector=await bge_m3.embed(intent.original_message),
        query_filter={"user_id": str(user_id)},
        limit=3
    )
    
    # 检索全局相似(不限用户)
    global_similar = await qdrant.search(
        collection_name="workflow_traces",
        query_vector=await bge_m3.embed(intent.original_message),
        query_filter={"user_satisfaction": {"gte": 4}},   # 只看高质量任务
        limit=5
    )
    
    return RAGContext(
        user_history=user_history,
        global_similar=global_similar
    )
```

主编排的意图理解 prompt 加入这部分上下文(只取摘要):

```
你是「有了」的总裁助理...

# 相似历史任务参考
该用户最近做过类似任务:
- 2 周前:做"复利效应"科普视频(抖音,60s)→ 满意度 5
- 1 周前:做"AI 新闻解读"视频(视频号,90s)→ 满意度 4

全局相似的高质量任务:
- 用户 X 做过类似选题,选了"快节奏 + 强钩子"风格,效果好
...
```

### 3.4 隐私

- 工作流轨迹**默认沉淀**(用户协议告知)
- 用户可在设置里关闭"用我的数据改进系统"
- 关闭后:
  - 信号 1 不沉淀到 Qdrant 全局可检索池(但仍在用户自己的 history)
  - 信号 3 / 4 不沉淀
- 全局检索时**自动脱敏**(去用户 ID、产物中的个人信息、上传素材)

### 3.5 成本

- BGE-M3 自托管,每次 embedding ~ 5ms,几乎免费
- Qdrant 存储 1024 维 × 100 万 traces ≈ 4 GB,3 节点集群够
- 单次 RAG 查询 ~ 30ms,加到主编排意图理解 < 5% 延迟

## 四、信号 2:用户偏好向量

### 4.1 升级理由

v1.0 偏好机制是**标量计数器**:

```json
{
  "anti_fraud_audience": {"value": "城市老人", "confidence": 0.7, "consecutive_count": 2}
}
```

问题:

1. 字段维度太多,每个独立 → 没有"偏好画像"
2. 无法处理"相似选项"(用户选过"快节奏",新选项"高节奏"应该也偏向)
3. 无法跨 Skill 迁移(短视频偏好不影响电商图)

### 4.2 升级方案:256 维向量

每个用户一个 `preference_vec VECTOR(256)`,用 **bge-base-zh-v1.5** 微调后的小模型生成。

向量空间设计:

| 维度组 | 范围 | 含义 |
|-------|------|------|
| 0-63 | 风格 | 简约 / 强烈 / 自然 / 卡通 / 写实... |
| 64-127 | 节奏 | 快 / 慢 / 平稳 / 跌宕 |
| 128-191 | 受众 | 年龄 / 性别 / 兴趣类 / 专业度 |
| 192-255 | 平台 | 抖音 / 小红书 / 视频号 / B 站 / 通用 |

### 4.3 更新机制

每次用户做选择:

```python
async def update_preference_vec(user_id: UUID, choice: UserChoice):
    """
    用户做了一次选择(澄清回答 / HITL 决定)→ 更新偏好向量。
    
    算法:加权移动平均
    new_vec = (1 - alpha) * old_vec + alpha * choice_vec
    
    alpha 随用户使用次数衰减:
    - 前 10 次任务:alpha = 0.3(快速学习)
    - 11-50 次:alpha = 0.1(稳定调整)
    - 50+ 次:alpha = 0.05(微调)
    """
    user_pref = await get_user_preferences(user_id)
    choice_vec = await bge_base_zh.embed(choice.semantic_label)
    
    alpha = compute_alpha(user_pref.total_choices)
    new_vec = (1 - alpha) * user_pref.preference_vec + alpha * choice_vec
    new_vec = normalize(new_vec)
    
    await pg.execute(
        "UPDATE user_preferences SET preference_vec = $1, total_choices = total_choices + 1, preference_updated_at = NOW() WHERE user_id = $2",
        [new_vec, user_id]
    )
```

### 4.4 用途:候选选项动态排序

澄清生成器在给选项时,**按用户偏好向量排序**:

```python
async def generate_clarification(field, user_id):
    options = field.options                     # 静态选项列表
    user_vec = await get_user_preference_vec(user_id)
    
    # 每个选项也有个语义向量(预计算缓存)
    option_vecs = [await get_option_vec(opt) for opt in options]
    
    # 按余弦相似度排序
    ranked = sorted(
        zip(options, option_vecs),
        key=lambda x: cosine_similarity(user_vec, x[1]),
        reverse=True
    )
    
    return Clarification(
        type="single_select",
        field=field.name,
        options=[opt for opt, _ in ranked],
        default_index=0   # 排序后第一个就是推荐
    )
```

### 4.5 冷启动

新用户(`total_choices < 3`):

- 进群时弹一个 4 题问卷(平台 / 风格 / 节奏 / 受众)→ 生成种子向量
- 不愿意填可跳过 → 用群类型默认向量(短视频群默认偏抖音 + 快节奏)

### 4.6 用户控制

设置页:**"偏好画像"**

- 可视化展示当前偏好向量(转译为人话:"你偏好抖音 + 快节奏 + 知识科普")
- 用户可手动调整:点 "我其实更喜欢小红书风格" → 系统直接覆盖向量某个维度
- 重置选项

## 五、信号 3:失败 → Reflexion → prompt 改进

### 5.1 触发条件

任务出现以下情况之一,自动跑 Reflexion pipeline:

1. Agent handler 返回 `failed`(模型 API 错误 / 工具失败 / 超时)
2. 质量校验 `quality_failure`(image_quality_check 评分 < 0.5)
3. 用户在 HITL gate 选 [都不要,重新生成] 或 [回到第 N 步重做](中断 C)
4. 任务完成后用户给满意度 ≤ 2 分

### 5.2 Reflexion Pipeline

用 Claude Sonnet 4.6 分析失败原因,提出 prompt 改进:

```python
async def reflexion_pipeline(task: Task, failure: FailureInfo):
    
    # 1. 收集上下文
    context = {
        "skill_id": task.skill_id,
        "step_id": failure.step_id,
        "original_prompt": failure.rendered_prompt,
        "user_input": task.collected_fields,
        "agent_response": failure.agent_response,
        "failure_mode": failure.type,
        "user_feedback": task.user_feedback if task.user_feedback else None
    }
    
    # 2. 让 Sonnet 分析
    analysis = await router.complete(
        task_type="reflexion",
        messages=[
            {"role": "system", "content": REFLEXION_SYSTEM_PROMPT},
            {"role": "user", "content": json.dumps(context, ensure_ascii=False)}
        ],
        routing_hints={"primary": "claude-sonnet-4-6"}
    )
    
    # analysis 包含:
    # - failure_root_cause: "..."
    # - prompt_section_to_improve: "..."  (具体哪部分需要改)
    # - proposed_change: "..."           (具体怎么改)
    # - expected_improvement: "..."       (预期效果)
    
    # 3. 写入候选表
    await pg.execute(
        """INSERT INTO prompt_improvement_candidates 
           (prompt_name, failure_task_id, root_cause, proposed_change, expected_improvement, status)
           VALUES ($1, $2, $3, $4, $5, 'pending')""",
        [...]
    )
```

`REFLEXION_SYSTEM_PROMPT`:

```
你是「有了」产品的 prompt 改进专家。

任务失败了,请分析:
1. 失败的根本原因(prompt 的哪个部分导致的?)
2. 应该改 prompt 的哪一部分
3. 改成什么样
4. 预期效果

输出 JSON,不要任何其他文字。
```

### 5.3 人工审核

每周一次,产品 + 算法工程师审 `prompt_improvement_candidates` 表:

- 看候选改进
- 决定是否合入 `app/config/prompts.py`
- 合入时改 prompt 版本号(`ORCHESTRATOR_INTENT_PROMPT_VERSION`)
- 灰度 10% 流量观察 24 小时
- 全量 / 回滚

### 5.4 自动化(V1.5)

V1.5 评估**自动 prompt 优化**(无人工审核):

- 候选改进经过 ML 评估器(基于 prompt 回归测试集打分)
- 评分 > 阈值自动进灰度
- 灰度数据驱动决策

但 V1 必须人工审核——LLM 改 prompt 仍有概率引入新 bug。

## 六、信号 4:高质量任务 → Skill 草稿

### 6.1 触发条件

任务满足:

1. 用户满意度 ≥ 4 分
2. **流程偏离平台预置 Skill**(用户做了非常规的中断 / 选择 / 调整)
3. 用户明确表达正面反馈(中断 H 类)

### 6.2 Skill Drafter Pipeline

```python
async def skill_drafter_pipeline(task: Task):
    
    # 1. 提取 trace
    trace = await load_workflow_trace(task.id)
    
    # 2. 让 Sonnet 起草 Skill YAML
    draft_yaml = await router.complete(
        task_type="skill_drafting",
        messages=[
            {"role": "system", "content": SKILL_DRAFTER_PROMPT},
            {"role": "user", "content": json.dumps(trace, ensure_ascii=False)}
        ],
        routing_hints={"primary": "claude-sonnet-4-6"}
    )
    
    # 3. 校验 YAML schema
    try:
        skill = Skill.from_yaml(draft_yaml)
    except ValidationError as e:
        logger.warning(f"Drafted skill invalid: {e}")
        return
    
    # 4. 写入 skill_drafts 表
    await pg.execute(
        """INSERT INTO skill_drafts 
           (user_id, source_task_id, draft_yaml, name, description, status)
           VALUES ($1, $2, $3, $4, $5, 'draft')""",
        [task.user_id, task.id, draft_yaml, skill.name, skill.description]
    )
    
    # 5. 通知用户
    await chat_service.send_assistant_message(
        conversation_id=task.conversation_id,
        content=f"""你刚才这个工作流我学到了!
        我把它整理成了一个 Skill 草稿:**{skill.name}**
        
        要发布让别人也能用吗?
        [发布]  [先保存]  [我自己改改]"""
    )
```

### 6.3 用户路径(V1.5 创作者飞轮)

```
用户做高质量任务 → 系统自动起 Skill 草稿
    ↓
通知用户 [发布] [保存] [改改]
    ↓ 用户点 [发布]
进入"我的 Skill"页面 → 编辑器(可改 YAML)
    ↓ 用户确认
平台审核(质量 + 不含违规 prompt)
    ↓ 通过
Skill 上架技能市场,用户成为创作者
    ↓
其他用户订阅使用 → 创作者获得分润(V2)
```

V1 上线时,这个流程是**关闭的**。V1.5 开放给少量内测创作者(50 人种子)。V2 开放给所有用户。

## 七、隐私与合规

### 7.1 用户协议必须明确

`产品协议 §X 数据使用`:

> 你使用「有了」时产生的工作流数据(对话、字段选择、HITL 决定、最终满意度)将用于:
> 1. 改进系统(prompt 优化、Skill 推荐)
> 2. 个性化你的体验(偏好画像)
> 3. 你授权的话,匿名化后用于训练
>
> 你可以在设置里关闭"用我的数据改进系统",关闭后:
> - 不再用于全局训练
> - 仅用于你自己的个性化
> - 已收集的数据保留(用于已上线的功能),不再增加

### 7.2 数据脱敏

全局检索 / 训练前必须脱敏:

- 用户 ID → 不可逆 hash
- 上传素材 → 不进训练集(只看流程,不看具体素材)
- 文本中的姓名 / 电话 / 地址 → regex 匹配后替换为 `[人名]` / `[电话]` 等
- 工作流概要 → LLM 重写为通用描述

### 7.3 删除权(GDPR / 个人信息保护法)

用户删除账号时:

- 立即删除 `users` / `user_preferences` 表的记录
- 24 小时内异步删除 `messages` / `tasks` / `task_steps` / `artifacts` / `workflow_traces` / `hitl_gates` / `skill_drafts`
- Qdrant 向量删除
- OSS 对象删除(异步,可能 7 天)

### 7.4 监管合规

中国市场:

- 算法备案(国家网信办 算法推荐管理规定)
- 生成式 AI 服务备案
- 内容安全审核(用户上传 + 模型输出都过阿里云内容安全)
