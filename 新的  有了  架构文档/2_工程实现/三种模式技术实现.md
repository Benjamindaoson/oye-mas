# 三种模式技术实现

**版本**:v3.0(对应 ADR-014;取代 ADR-008 的两种独立群方案)
**日期**:2026-05-05
**面向**:后端 / 前端 / 智能体团队工程师
**地位**:Plan / Ask / Auto 三模式同群切换的工程落地。产品定义请看 [三种模式产品定义](../1_原理篇/三种模式产品定义.md)。

**v3.0 关键变更**:
- ✨ **三种模式同群内切换**(ADR-014)取代两种独立群
- ✨ `conversation_links` 表 / `parent_conversation_id` 字段废弃
- ✨ 新增 `conversations.work_mode`(plan / ask / auto)
- ✨ 新增 `switch-work-mode` API 取代 `derive-work-group` / `back-to-discuss`
- ✨ Brief 维护逻辑改:同群内跨模式延续(不再"两群共享 pool")

---

## 一、核心工程设计

**一个群 = 一个对话单位 + 三种工作方式可中途切换**——这是核心设计。

- **用户视角**:进群后选模式 / 群顶部 chip 切换 / 总裁助理识别意图自动切
- **工程视角**:`conversations.work_mode` 字段记录当前模式,LangGraph state 不重启
- **Brief**:每个群一个 ContextPool(一对一),不再跨群共享

## 二、数据模型

### 2.1 整体关系图

```
User
 │
 ├── 创建多个 Conversation(每个就是"群"或"主会话")
 │     │
 │     ├── Conversation(mode=main_session, work_mode=plan)  ──► 主会话
 │     │     │ context_pool_id ──────────► ContextPool
 │     │     └── Messages(独立)
 │     │
 │     ├── Conversation(mode=group, work_mode=plan|ask|auto) ──► 通用工作群
 │     │     │ context_pool_id ──────────► ContextPool(每群一个)
 │     │     │ skill_id = null
 │     │     └── Messages(独立)
 │     │
 │     └── Conversation(mode=group, work_mode=plan|ask|auto) ──► 专门工作群
 │           │ context_pool_id ──────────► ContextPool(每群一个)
 │           │ skill_id = 反诈视频 / 电商详情图
 │           └── Messages(独立)
 │
 └── ContextPool(每群一个,不再跨群共享)
       ├── intent
       ├── brief
       ├── reference_artifact_ids[]
       ├── linked_conversation_ids[]
       └── user_preferences_snapshot
```

### 2.2 Conversation 表(v3.0)

```sql
CREATE TABLE conversations (
    id              UUID PRIMARY KEY,
    user_id         UUID NOT NULL,
    name            VARCHAR(100) NOT NULL,
    mode            VARCHAR(20) NOT NULL CHECK (mode IN ('main_session','group','private_chat')),
    -- v3.0:简化,不再有 'discuss'/'work',group 是唯一的"群"类型
    
    work_mode       VARCHAR(10) CHECK (work_mode IN ('plan','ask','auto')),
    -- v3.0 新增:工作模式,group 类型时必填
    -- main_session 也用 work_mode(默认 plan,因为主会话主要做元操作 + 讨论)
    
    avatar_style    VARCHAR(50),
    
    context_pool_id UUID,
    -- v3.0:每群一个 pool,不跨群共享
    
    skill_id        UUID,
    -- 专门群挂 skill_id;通用群和主会话为 null
    
    -- v3.0 删除:parent_conversation_id(不再衍生)
    
    status          VARCHAR(20) DEFAULT 'active' CHECK (status IN ('active','paused','archived','deleted')),
    
    last_message_at TIMESTAMPTZ,
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW(),
    
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (context_pool_id) REFERENCES context_pools(id),
    FOREIGN KEY (skill_id) REFERENCES skills(id)
);

CREATE INDEX idx_conv_user_mode ON conversations(user_id, mode);
CREATE INDEX idx_conv_user_work_mode ON conversations(user_id, work_mode) WHERE mode='group';
CREATE INDEX idx_conv_context_pool ON conversations(context_pool_id);
CREATE INDEX idx_conv_last_msg ON conversations(user_id, last_message_at DESC);
```

**v2 → v3 迁移**:

```sql
-- 1. 把 mode='discuss' 的迁移为 mode='group', work_mode='plan'
UPDATE conversations SET mode='group', work_mode='plan' WHERE mode='discuss';
-- 2. 把 mode='work' 的迁移为 mode='group', work_mode='auto'
UPDATE conversations SET mode='group', work_mode='auto' WHERE mode='work';
-- 3. 主会话 work_mode 默认 plan
UPDATE conversations SET work_mode='plan' WHERE mode='main_session';
-- 4. 删 parent_conversation_id 列
ALTER TABLE conversations DROP COLUMN parent_conversation_id;
-- 5. drop conversation_links 表
DROP TABLE conversation_links;
```

### 2.5 Artifacts 表(关键变化)

产物用 context_pool_id 关联,而不是 conversation_id——这样讨论群和工作群可以共享产物。

```sql
CREATE TABLE artifacts (
    id              UUID PRIMARY KEY,
    context_pool_id UUID NOT NULL,
    -- 关键:用 pool 关联,而不是 conversation
    
    source_conversation_id UUID NOT NULL,
    -- 哪个群产出的(用于显示来源)
    
    type            VARCHAR(30) NOT NULL,
    -- 'reference_image' / 'final_video' / 'script' / 'xlsx' / etc.
    
    is_reference    BOOLEAN DEFAULT FALSE,
    -- 标记是否为讨论中的"参考图"(快速、低质量)
    
    is_final        BOOLEAN DEFAULT FALSE,
    -- 标记是否为工作群的"最终交付物"
    
    reference       TEXT NOT NULL,
    -- OSS URL
    
    metadata        JSONB,
    
    created_at      TIMESTAMP DEFAULT NOW(),
    
    FOREIGN KEY (context_pool_id) REFERENCES context_pools(id),
    FOREIGN KEY (source_conversation_id) REFERENCES conversations(id)
);

CREATE INDEX idx_artifact_pool ON artifacts(context_pool_id);
CREATE INDEX idx_artifact_source ON artifacts(source_conversation_id);
```

### 2.6 Messages 表

```sql
CREATE TABLE messages (
    id              UUID PRIMARY KEY,
    conversation_id UUID NOT NULL,
    
    role            VARCHAR(20),
    -- 'user' / 'assistant' / 'agent_1' / 'agent_2' / 'agent_3' / 'agent_4'
```

## 五、ConversationService 关键方法

```python
class ConversationService:
    
    async def create_conversation(
        self,
        user_id: UUID,
        mode: Literal["main_session", "group", "private_chat"],     # v3.0
        work_mode: Literal["plan", "ask", "auto"] = "plan",          # v3.0 新增
        skill_id: Optional[UUID] = None,
        name: Optional[str] = None
    ) -> Conversation:
        """
        创建新群(v3.0 ADR-014 三模式同群)。
        
        逻辑:
        1. 创建新 context_pool(每群一个 pool,不跨群共享)
        2. 创建 conversation,默认 work_mode=plan
        """
        
        pool = await context_pool_service.create(user_id=user_id)
        
        if not name:
            name = await self._generate_default_name(mode, skill_id)
        
        conv = Conversation(
            id=uuid4(),
            user_id=user_id,
            name=name,
            mode=mode,
            work_mode=work_mode,                                     # v3.0
            context_pool_id=pool.id,
            skill_id=skill_id,
            avatar_style=self._select_avatar_style(mode, skill_id),
            status="active"
        )
        await db.save(conv)
        
        await ws.emit("conversation_created", conv.to_dict())
        await self._send_welcome_message(conv)
        
        return conv
    
    async def switch_work_mode(
        self,
        conversation_id: UUID,
        target: Literal["plan", "ask", "auto"],
        triggered_by: Literal["user", "orchestrator"] = "user"
    ) -> Conversation:
        """同群内切换工作模式(v3.0 ADR-014,替代 derive_work_group / back_to_discuss)"""
        
        conv = await self.get(conversation_id)
        old_mode = conv.work_mode
        
        if old_mode == target:
            return conv  # no-op
        
        # 如果当前 Auto 模式有任务在执行,先暂停
        if old_mode == "auto" and conv.has_running_task():
            await langgraph_api.update_state(
                thread_id=conv.task_id,
                values={"status": "paused"}
            )
        
        # 写入 mode_switch_log
        await db.save(ModeSwitchLog(
            conversation_id=conv.id,
            from_mode=old_mode,
            to_mode=target,
            triggered_by=triggered_by
        ))
        
        # 更新 conversation
        conv.work_mode = target
        await db.save(conv)
        
        # 推系统消息 + WS 事件
        await chat_service.send_system_message(
            conv.id,
            f"─── 总裁助理 切到 {MODE_DISPLAY[target]} ───"
        )
        await ws.emit("work_mode_changed", {
            "conversation_id": str(conv.id),
            "from": old_mode,
            "to": target
        })
        
        # 切到 Auto 时,如果有 brief 则自动启动 Skill
        if target == "auto" and conv.skill_id:
            await orchestrator.start_skill_with_brief(conversation=conv, skill_id=conv.skill_id)
        
        return conv
```

## 六、ContextPoolService 关键方法

```python
class ContextPoolService:
    
    async def create(self, user_id: UUID) -> ContextPool:
        """创建空的上下文池"""
        pool = ContextPool(
            id=uuid4(),
            user_id=user_id,
            intent={},
            brief={"完成度": 0.0, "字段": {}, "决策日志": []},
            reference_artifact_ids=[],
            user_preferences_snapshot=await self._snapshot_user_prefs(user_id)
        )
        await db.save(pool)
        return pool
    
    async def update_brief(
        self,
        pool_id: UUID,
        new_message: str,
        conversation_id: UUID
    ):
        """讨论群里防抖批量更新 brief"""
        pool = await self.get(pool_id)
        recent_history = await msg_service.recent(conversation_id, limit=5)
        
        new_brief = await llm_update_brief(
            current_brief=pool.brief,
            new_message=new_message,
            recent_history=recent_history
        )
        
        old_completion = pool.brief.get("完成度", 0)
        pool.brief = new_brief
        pool.updated_at = now()
        await db.save(pool)
        
        # 跨过 0.8 阈值 → 主动建议
        if new_brief["完成度"] >= 0.8 and old_completion < 0.8:
            await self._suggest_work_group(conversation_id, pool)
    
    async def add_reference_artifact(self, pool_id: UUID, artifact_id: UUID):
        """讨论群产出参考图,加进池"""
        pool = await self.get(pool_id)
        pool.reference_artifact_ids.append(artifact_id)
        await db.save(pool)
    
    async def cleanup_orphan_pools(self):
        """V1 用定时任务清理:讨论群+所有衍生工作群都删除后,孤儿池清理"""
        sql = """
        DELETE FROM context_pools
        WHERE id NOT IN (
            SELECT DISTINCT context_pool_id FROM conversations
            WHERE status != 'deleted' AND context_pool_id IS NOT NULL
        )
        AND created_at < NOW() - INTERVAL '7 days'
        """
        await db.execute(sql)
```

## 七、BriefBuilder(讨论群产出 brief)

### 7.1 防抖批量调度器

**触发条件**(满足任一即更新):

1. 用户停顿 5 秒(打字结束防抖)
2. 累计 3 条新消息未更新
3. 用户主动点 [总结一下]
4. 总裁助理被 @ 时强制更新

```python
async def schedule_brief_update(pool_id, new_message, conversation_id):
    counter = await redis.incr(f"brief_pending:{pool_id}")
    
    if counter >= 3:
        await _do_update(pool_id, conversation_id)
        await redis.delete(f"brief_pending:{pool_id}")
        await scheduler.cancel(f"brief_debounce:{pool_id}")
    else:
        # 5 秒防抖
        await scheduler.schedule_once(
            f"brief_debounce:{pool_id}",
            delay=5,
            callback=lambda: _do_update(pool_id, conversation_id)
        )

async def _do_update(pool_id, conversation_id):
    await context_pool_service.update_brief(
        pool_id=pool_id,
        new_message=None,  # batch mode: 用最近 N 条消息合并更新
        conversation_id=conversation_id
    )
```

### 7.2 LLM 更新 brief

```python
async def llm_update_brief(
    current_brief: dict,
    new_message: str,
    recent_history: list
) -> dict:
    
    prompt = f"""
你是「有了」总裁助理的 brief 维护器。

任务:基于对话上下文,持续更新一个"需求 brief"。

当前 brief:
{json.dumps(current_brief, ensure_ascii=False, indent=2)}

最近对话:
{format_history(recent_history)}

用户最新消息:
{new_message}

请更新 brief。规则:
1. 字段格式:{{"产品": "...", "受众": "...", "风格": "...", ...}}
2. 只填用户明确提到的内容,不要猜
3. 完成度评估:0.0(空白)→ 1.0(全部清楚)
4. 决策日志:记录关键决策(如"用户选了自然质朴风")

输出 JSON:
{{
  "完成度": 0.0-1.0,
  "字段": {{...}},
  "决策日志": [...]
}}
"""
    
    response = await router.complete(
        task_type="brief_update",
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"},
        timeout=2,
        routing_hints={
            "primary": "deepseek-v4-flash",
            "fallback": ["claude-haiku-4-5"]
        }
    )
    
    return json.loads(response.content)
```

```python
async def start_skill_with_brief(conversation, skill_id):
    """工作群启动时,用 brief 自动填字段"""
    pool = await context_pool_service.get(conversation.context_pool_id)
    skill = await skill_service.get(skill_id)
    
    brief_fields = pool.brief.get("字段", {})
    validation = input_validator.validate(
        skill, intent=None, user_prefs=pool.user_preferences_snapshot, brief=pool.brief
    )
    
    # 通知用户 brief 已自动填充
    if brief_fields:
        await chat_service.send_assistant_message(
            conversation_id=conversation.id,
            content=f"已自动应用讨论群里聊清楚的:\n{format_filled(brief_fields)}\n",
        )
    
    if validation.missing:
        clarification = await clarification_generator.generate(validation.missing)
        await chat_service.send_clarification(conversation.id, clarification)
    else:
        await orchestrator.start_workflow(
            conversation=conversation,
            skill=skill,
            inputs=validation.filled
        )
```

### 9.2 跨群跳转

**讨论群 → 工作群**:用户点 [拉工作群开始制作]

1. 前端发请求 `POST /api/conversations/{讨论群id}/derive-work-group`
2. 后端创建工作群,返回新 conversation
3. 前端左栏插入新工作群
4. 前端自动切换到工作群

**工作群 → 讨论群**:用户点 [回讨论群]

1. 前端发请求 `POST /api/conversations/{工作群id}/back-to-discuss`
2. 后端检查 parent_conversation,有则跳转,无则新建讨论群
3. 工作群任务标记 paused

## 十、关键场景的端到端时序

### 10.1 场景 A:模糊型用户走完整链路

```
T+0       用户在主会话:"我想做电商详情图"
T+0.3     主编排意图理解(deepseek-v4-flash)
          intent: { type:create_task, scenario:ecommerce_detail, confidence:0.92 }
T+0.4     主编排发送模式选择
          [讨论模式] [干活模式] [我也不知道]
T+10      用户点 [讨论模式]
T+10.1    后端 ConversationService.create_conversation(mode=discuss)
          - 创建 ContextPool
          - 创建 Conversation
          - 总裁助理 + 研究员 + 设计师 在场
T+10.5    跳转到讨论群,总裁助理出场
          "你想做什么样的电商图?能告诉我产品和受众吗?"

[讨论 5 分钟]

T+5min    讨论中,brief 实时更新(防抖批量):
          { 产品:有机草莓, 受众:城市妈妈, 风格:自然质朴, 完成度:0.85 }
T+5min    BriefBuilder 检测到完成度 ≥ 0.8(且之前 < 0.8)
T+5min    总裁助理主动:
          "已经聊清楚了:有机草莓,城市妈妈,自然质朴风。
           按这个开干?[拉工作群开始制作] [还要再聊]"
T+5min    用户点 [拉工作群开始制作]
T+5min    后端 derive_work_group(discuss_id, skill=电商详情图)
          - 创建工作群,共享 context_pool_id
          - 建立 conversation_link
          - 工作群读取 brief,自动填充 Skill inputs
          - 跳过已填字段的澄清
T+5.5min  跳转到工作群,主编排:
          "已自动应用讨论中聊清楚的需求:

```

## 十一、用户偏好沉淀(扩展)

### 11.1 模式偏好

| 用户行为 | 偏好状态 |
|---------|---------|
| 第 1 次新建群,选 X 模式 | 不影响下次 |
| 连续 2 次同选 X 模式 | 下次默认显示 X(仍要点确认)|
| 连续 3 次同选 X 模式 | 下次直接进 X 模式(跳过选择)|
| 之后选了不同模式 | 偏好重置(回到第 1 次状态)|

### 11.2 存储

`user_preferences` 表:

```json
{
  "default_mode": {
    "value": "work",
    "confidence": 1.0,
    "consecutive_count": 3,
    "last_updated": "..."
  },
  "short_video_platform": {
    "value": "抖音",
    "confidence": 1.0,
    "consecutive_count": 3
  }
}
```

> **v2.0 升级**:除了上述标量计数器,还有 `preference_vec VECTOR(256)` 列(bge-base-zh-v1.5 微调),持续更新捕获 4 维度偏好(风格/节奏/受众/平台)。详见 [数据飞轮设计](../3_决策记录/数据飞轮设计.md)。

## 十三、已知工程限制

### 13.1 brief 实时更新成本

- 每条消息都更新 brief 会消耗 LLM 调用
- 优化方案(已落地):**5 秒防抖 + 累计 3 条触发** 二选一
- 预估成本:讨论 5 分钟 ≈ 10-15 条消息 → 触发 3-5 次 brief 更新 ≈ $0.003

### 13.2 上下文池清理

- 讨论群和所有衍生工作群都删除后,context_pool 应该自动清理
- V1:每天定时任务 `cleanup_orphan_pools`,7 天宽限期
- V2:实时清理(订阅 conversation 删除事件)

### 13.3 跨群消息查找

- 主会话查询时(如"上次电商图聊到哪了"),需要跨群搜索
- V1:简单按 user_id + 关键词 SQL 查询
- V2:用向量检索

### 13.4 工作群暂停后跳回讨论的状态恢复

- 工作群标记 paused 后,LangGraph state 完整保留
- 用户回工作群时,可恢复执行
- 用户回讨论群讨论改了 brief 后,工作群恢复时主编排需要判断 brief 变化是否影响已完成的步骤(V2 中断 C 类)

## 十五、关键测试场景

- 模糊型用户:进讨论群 → 衍生工作群 → 完成
- 明确型用户:进工作群 → 完成
- 中途切换:工作群 → 讨论群 → 回工作群
- 一对多:一个讨论群衍生 3 个工作群
- 偏好沉淀:连续 3 次选同模式后默认行为
- 配额检查:讨论群不消耗任务配额
- 数据一致性:产物在两个群都能看到
- 防抖逻辑:5 秒内连续打字只更新 1 次 brief

## 十六、监控指标

- 模式选择分布(讨论 vs 干活 vs 不确定)

- 工作群跳回讨论群的频率

- 平均讨论 → 工作的总耗时
